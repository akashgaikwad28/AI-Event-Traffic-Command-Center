# ML Pipeline Architecture

The ML pipeline manages the model training, evaluation, and logging lifecycle. It is primarily driven by the training orchestrator in `ml_pipeline/` and MLOps tools in `mlops/`.

## 1. Training Orchestration (`ml_pipeline/training/train_models.py`)
- The `ModelTrainer` class loads the processed Parquet dataset generated by the Data Pipeline.
- Selects core features: `latitude`, `longitude`, `hour_of_day`, `day_of_week`, `is_weekend`, `is_peak_hour`, `priority_encoded`, `is_closed`.
- Trains three primary models:
  - **Congestion Proxy Model**: An XGBoost Regressor predicting a `congestion_proxy_score`.
  - **Deployment Load Model**: A Random Forest Classifier predicting `deployment_load_class` (e.g., "LOW", "HIGH").
  - **Response Time Model**: An XGBoost Regressor predicting `resolution_time_minutes` for completed incidents.
- Exports the trained models via `joblib` as artifacts (`congestion_model.pkl`, `deployment_model.pkl`, `response_time_model.pkl`) to `backend/app/ai/artifacts`.

## 2. Model Evaluation
- Model metrics such as RMSE, MAE, R² (for regressions), and Accuracy, Precision, Recall, F1 (for classifications) are calculated using the `ModelEvaluator`.
- The `ModelVisualizer` generates visual assets like confusion matrices and feature importance graphs, saving them to `reports/figures/`.

## 3. MLOps and MLflow Tracking (`mlops/mlflow/tracker.py`)
- The pipeline seamlessly integrates MLflow tracking configured with **DagsHub**.
- Authentication and initialization are handled using a DagsHub user token, pointing the MLflow tracking URI to the remote repository.
- `MLFlowTracker` logs all training parameters (e.g., used feature sets), evaluation metrics, and registers the resulting XGBoost and Sklearn models directly into the MLflow model registry.

## 4. Model Registry (`mlops/model_registry/registry_manager.py`)
- In addition to MLflow, the `RegistryManager` maintains a local system registry (e.g., `model_registry.json`).
- Automatically tracks and promotes trained models to the "Production" state, capturing crucial metadata such as model version, required input features, and final validation metrics for the backend to consume safely in inference tasks.
